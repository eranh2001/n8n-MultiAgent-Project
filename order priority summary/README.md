# Order Priority Summary and Report

A workflow automation that reads a list of orders, classifies each one into a priority tier (High, Medium, Low) based on its value, and produces a clean prioritized summary that is emailed as a report. Built in n8n with no AI, this project is pure workflow logic, branching, and data transformation.

Built on n8n Cloud, with a JavaScript Code node for the summary and Gmail for delivery.

---

## The Problem

When a business receives a list of orders, not all of them are equally important. High-value orders often need attention first. Manually sorting orders, grouping them by value, and writing up a summary is repetitive and easy to get wrong.

This workflow does it automatically. It reads the orders, decides each one's priority based on its value, and delivers a single ranked summary report, so the team can see at a glance what matters most.

---

## How It Works

```
Trigger
   │
   ▼
Get rows from Google Sheets   (the orders)
   │
   ▼
Sort by amount
   │
   ▼
Switch  (routes each order by value)
   ├── High    ──▶ Edit Fields: tag priority "High"
   ├── Medium  ──▶ Edit Fields: tag priority "Medium"
   └── Low     ──▶ Edit Fields: tag priority "Low"
                          │
                          ▼
                       Merge  (combine all three groups back together)
                          │
                          ▼
                  Code node  (build the prioritized summary)
                          │
                          ▼
                  Send email  (deliver the report)
```

1. **Read and sort.** The workflow reads the orders from Google Sheets and sorts them by amount.
2. **Classify by value.** A Switch node routes each order into one of three branches based on its amount: High, Medium, or Low.
3. **Tag the priority.** Each branch passes through its own Edit Fields node, which adds a `priority` label to the order matching that branch.
4. **Merge.** The three branches are combined back into one list, now with every order tagged by priority.
5. **Summarize.** A JavaScript Code node sorts the orders by priority and builds a readable summary line for each one, for example `[HIGH] Mike, Combo Bundle, $750`.
6. **Deliver.** The finished summary is emailed out as a report.

---

## Why It Is Built This Way

- **The priority is generated by the workflow, not assumed from the data.** The source orders only contain customer, product, and amount. There is no priority field. The workflow itself derives the priority by routing each order through the Switch by value and tagging it, so the classification logic lives in the automation and is easy to adjust.
- **Branch, then merge.** Splitting the orders into three branches lets each priority group be handled separately, and the Merge node brings them back into a single list for the final summary. This is a common and reliable pattern for "categorize, then recombine."
- **A Code node for the final formatting.** The summary needs custom logic (sorting by priority, building a formatted line per order). A Code node handles this cleanly where the visual nodes alone would be awkward.

## A Problem I Debugged

The Switch routed orders into priority tiers by amount, and the Medium tier used a range condition (amount between 200 and 599). It kept failing, because the amount arrived from Google Sheets as text rather than a number, so the numeric range comparison did not evaluate correctly.

I fixed it by explicitly converting the value to a number on both sides of the comparison:

```
{{ Number($json.amount) >= 200 && Number($json.amount) <= 599 }}
```

This is a common real-world issue: data from external sources often comes in as text even when it looks numeric, so any time the workflow does math or range comparisons, the value needs to be converted to a number first.
